Why a Virtual Machine?

  • Decouple compilation (expensive, once) from execution (fast, millions of shots)
  • Compact bytecode is cache-friendly — 32-byte aligned instructions
  • Optimizer can fuse, reorder, and specialize ops before execution
  • Same bytecode can run on CPU or GPU backends

Bytecode Opcodes

The SVM instruction set

Frame Opcodes — O(1) Pauli Updates

These never touch the amplitude vector. Pure bitwise operations on p_x and p_z.

OpcodeAction on Pauli FrameCost
OP_FRAME_H swap p_x[i] ↔ p_z[i] 1 XOR + swap
OP_FRAME_S p_z[i] ^= p_x[i] 1 XOR
OP_FRAME_S_DAG p_z[i] ^= p_x[i] (same + phase) 1 XOR
OP_FRAME_CNOT p_x[t] ^= p_x[c]; p_z[c] ^= p_z[t] 2 XORs
OP_FRAME_CZ p_z[i] ^= p_x[j]; p_z[j] ^= p_x[i] 2 XORs
OP_FRAME_SWAP swap p_x[i] ↔ p_x[j] and p_z[i] ↔ p_z[j] 2 swaps

A QEC circuit with 10,000 Clifford gates → 10,000 bitwise ops. No floating-point math.

Array Opcodes — O(2k) Amplitude Operations

These operate on the amplitude vector v[0..2k-1] with strided loops.

Single-Qubit Array Ops
OP_ARRAY_H — Hadamard on active qubit
OP_ARRAY_S / OP_ARRAY_S_DAG — phase gate
OP_ARRAY_T / OP_ARRAY_T_DAG — T gate (π/8)
OP_ARRAY_ROT — arbitrary Rz(θ) rotation
OP_ARRAY_U2 — fused single-axis 2×2 unitary
Two-Qubit Array Ops
OP_ARRAY_CNOT — controlled-NOT
OP_ARRAY_CZ — controlled-Z
OP_ARRAY_SWAP — swap two active qubits
OP_ARRAY_U4 — fused 2-axis 4×4 unitary
Multi-Gate (Fused) Ops
OP_ARRAY_MULTI_CNOT — 64-bit bitmask of controls
OP_ARRAY_MULTI_CZ — 64-bit bitmask of targets

Strided Loop Pattern

// T gate on axis a among k active qubits:
uint64_t stride = 1ULL << a;
for (uint64_t i = 0; i < (1ULL << k); i++) {
  if (i & stride)
    v[i] *= exp(i*pi/4); // phase
}

Uses PDEP (BMI2) for branchless qubit indexing on x86.

Subspace Opcodes — Growing & Shrinking k

OP_EXPAND: k → k+1

When a dormant qubit (in |0⟩) needs to become active for a non-Clifford gate:

  1. Allocate new vector of size 2k+1
  2. Copy existing amplitudes to even-indexed slots
  3. Zero-fill odd-indexed slots (they correspond to |1⟩ of the new qubit)
  4. Increment active_k

OP_EXPAND_T — Fused Expand+T

Combines expand and T-gate in one pass. Avoids allocating the intermediate state. The optimizer's ExpandTPass produces these.

Visual: EXPAND doubles the vector

Measurement Opcodes

Four categories based on whether the measured qubit is dormant vs active and the measurement basis.

OpcodeQubit StateMeasurement TypeCost
OP_MEAS_DORMANT_STATIC Dormant Deterministic (known |0⟩ or |1⟩) O(1)
OP_MEAS_DORMANT_RANDOM Dormant Random 50/50 (X-basis of |0⟩) O(1)
OP_MEAS_ACTIVE_DIAGONAL Active Z-basis (diagonal) — Born rule on |v[i]|² O(2k)
OP_MEAS_ACTIVE_INTERFERE Active X/Y-basis (off-diagonal) — requires Hadamard first O(2k)
OP_SWAP_MEAS_INTERFERE Active Fused swap + interfering measurement O(2k)

After Measurement

  • Born rule: sample outcome with probability |amplitude|²
  • Collapse: zero out inconsistent amplitudes, renormalize
  • If qubit becomes deterministic → can REDUCE k (shrink vector)

Forced Variants

Each measurement opcode has a _FORCED variant for importance sampling. The outcome is predetermined from forced_record — tracks log-probability instead of sampling.

How Gates Transform State

Clifford vs non-Clifford execution paths

Clifford Gates: Pauli Frame Only

H gate on qubit i

p_x[i] ↔ p_z[i]

X ↔ Z under Hadamard

S gate on qubit i

p_z[i] ^= p_x[i]

X → Y (= iXZ) under S

CNOT (c → t)

p_x[t] ^= p_x[c]
p_z[c] ^= p_z[t]

X flows forward, Z flows back

All frame updates are single bit operations — no floating-point math, no memory allocation, no loops.

Non-Clifford: The T Gate

T = diag(1, eiπ/4)

Applies phase eiπ/4 to every amplitude where the target qubit is |1⟩.

Strided Loop over v[]

// T gate on active axis 'a' at rank k
const complex<double> w = exp(i * pi/4);
uint64_t half = 1ULL << k;
uint64_t stride = 1ULL << a;

for (uint64_t i = 0; i < half; i++) {
  // PDEP maps logical index to
  // physical index, skipping axis 'a'
  uint64_t lo = pdep(i, ~stride);
  v[lo | stride] *= w;  // |1> gets phase
  // v[lo] unchanged      // |0> stays
}

Cost: O(2k) complex multiplications

Visual: T on axis 1, k=3

EXPAND: Dormant → Active

When Does EXPAND Happen?

A non-Clifford gate targets a dormant qubit (one that is still in the tensor product |0⟩D). The qubit must become active before the gate can be applied.

Algorithm

  1. Allocate new_v[2k+1]
  2. For each index j in 0..2k-1:
    • new_v[2j] = old_v[j] (qubit in |0⟩)
    • new_v[2j+1] = 0 (qubit in |1⟩)
  3. Update active_k = k + 1
  4. Free old vector

Cost: O(2k) — one memcpy-like pass

EXPAND + Measurement Can REDUCE

Measurement Collapse → k−1

After measuring an active qubit, half the amplitudes become zero. If the remaining state is separable, the qubit returns to dormant and k decreases.

Measurement: Born Rule → Collapse

Step 1: Compute Probabilities

Diagonal (Z-basis)

// Sum |v[i]|^2 where target bit = 0
double p0 = 0;
for (uint64_t i = 0; i < (1<<k); i++)
  if (!(i & stride))
    p0 += norm(v[i]);
// p1 = 1 - p0 (or sum directly)

Interfering (X/Y-basis)

Must first apply an implicit Hadamard to convert X/Y eigenstates into Z eigenstates, then sum. More expensive due to complex additions.

Step 2: Sample & Collapse

Born Rule Sampling

// PRNG: Xoshiro256++
double u = rng.uniform();
bool outcome = (u > p0) ? 1 : 0;

State Collapse

// Zero out amplitudes inconsistent
// with the measurement outcome
for (uint64_t i = 0; i < (1<<k); i++)
  if (bit_of(i,a) != outcome)
    v[i] = 0;
// Renormalize via gamma

After collapse: if all non-zero amplitudes agree on qubit a, the qubit can be reduced back to dormant, shrinking k → k−1.

The Execution Loop

Dispatch, ISA selection, and parallelism

Computed-Goto Dispatch Loop

The SVM uses a threaded dispatch pattern via GCC/Clang computed goto (with a switch fallback for MSVC).

void execute_internal(const CompiledModule& program,
                      SchrodingerState& state) {
    // Build jump table: opcode -> label address
    static const void* dispatch_table[] = {
        [OP_FRAME_CNOT]  = &&L_FRAME_CNOT,
        [OP_ARRAY_H]     = &&L_ARRAY_H,
        [OP_EXPAND_T]    = &&L_EXPAND_T,
        [OP_NOISE_BLOCK] = &&L_NOISE_BLOCK,
        ...
    };
    const Instruction* pc  = program.bytecode.data();
    const Instruction* end = pc + program.bytecode.size();

    goto* dispatch_table[pc->opcode];   // first opcode

  L_ARRAY_H:
    exec_array_h(state, pc->axis_1);
    DISPATCH();   // ++pc; if (pc==end) return; goto*
    ...
}

Why computed goto? Eliminates branch predictor pollution from a central switch. Each label site has its own branch-target-buffer entry, giving near-zero dispatch overhead per opcode.

ISA Dispatch: Runtime CPUID Selection

Three copies of every kernel are compiled into separate translation units with different -m flags. At startup, CPUID selects the best one.

Parallel Loop Helpers

OpenMP threading engages only when active_k ≥ 18 (statevector ≥ 4 MB). Below that, a C++ if-guard bypasses the OpenMP runtime entirely -- no futex overhead.

parallel_stride_loop

2D waterfall for 1-qubit gates. Outer blocks of size 2*step, inner sweep at inner_stride. Parallelizes outermost dimension when ≥ 4 iterations.

for oi in 0..total/(2*step):   # OMP here
  for k in 0..step by stride:
    kernel(oi*2*step, k)

parallel_3d_stride_loop

3D waterfall for 2-qubit gates. Three nested dimensions: outer, mid, inner. Parallelizes the outermost level with ≥ 4 iterations; falls back inward.

for oi in 0..total/(2*step_hi): # OMP here
  for mi in 0..step_hi/(2*step_lo):
    for ki in 0..step_lo/stride:
      kernel(base, ki*stride)

parallel_for

Simple 1D loop over v[] indices

parallel_reduce

2 or 3 accumulators for measurement probability sums via OMP reduction

parallel_flat_loop

Flat strided loop for uniform array sweeps

Section 7

Sampling Modes

Six ways to query the quantum state

Shot-Based Sampling Modes

Mode 1: sample(shots)

Standard Monte Carlo. Run the full bytecode shots times, collect all measurement / detector / observable records. PRNG seeded once and streams forward across shots -- no per-shot reseeding.

Mode 2: sample_survivors(shots)

Post-selection aware. Runs all shots but keeps only those that pass every OP_POSTSELECT check. Returns passed_shots, logical_errors, and per-observable counts. With keep_records=false, zero array allocation -- counts only.

Mode 3: sample_k(shots, k)

Importance sampling. Forces exactly k fault events per shot (quantum noise + readout). Sites drawn from the conditional Poisson-Binomial distribution via a DP table sweep. Uniform-probability circuits get an O(k) Fisher-Yates sampler automatically.

All three share the same SchrodingerState + execute() path. The difference is how noise sites fire and which shots are kept.

Exact Computation Modes

Mode 4: basis_probabilities(masks)

Exact Born-rule probabilities |<x|psi>|² for given computational-basis states. No sampling, no noise. Unitary programs only (no measurements, feedback, or noise opcodes). Each basis mask is word-packed little-endian.

Mode 5: record_probabilities(records)

Exact log-probabilities for a batch of measurement records. Computes log P(record) under the compiled program, using forced-outcome kernels that read from forced_record[] instead of the PRNG and accumulate log-prob. Unreachable records return -DBL_MAX.

Mode 6: get_statevector()

Debug / validation oracle. Expands the factored state into a dense 2^n vector: embed |phi>_A, apply Pauli frame P, multiply by Clifford unitary U_C, scale by gamma. Capped at n ≤ 10 qubits (8 MB dense unitary matrix).

Which Sampling Mode to Use?

Debug only: get_statevector() is for validation of small circuits (n ≤ 10). Never use in production -- O(4^n) unitary expansion.

PRNG

Xoshiro256++ with SplitMix64 seeding

Xoshiro256++ PRNG

Why Xoshiro256++?

  • 32 bytes state (vs. MT19937's 2504 bytes) -- per-shot reseeding ~100x cheaper
  • Period: 2256 - 1
  • Pure bitwise math -- identical sequences across GCC / Clang / MSVC
  • Blackman & Vigna (2021), ACM Trans. Math. Softw.

53-bit Uniform Doubles

// CRITICAL: avoid std::uniform_real_distribution
// -- its output varies across compilers!
double random_double() {
  return (rng_() >> 11) * 0x1.0p-53;
}

Shift right 11 bits to keep the top 53 significant bits, then scale to [0, 1) with exact IEEE-754 representability.

Noise Handling

Skip-ahead gap sampling and noise blocks

Skip-Ahead Gap Sampling

Instead of rolling the RNG at every noise site, sample an exponential gap to jump directly to the next site that fires.

Naive Approach

for each site in 0..N_sites:
  if random() < site.probability:
    apply_pauli(site.channels)
// Cost: O(N_sites) per shot
// d=17 surface code: ~5000 sites

Gap Sampling

gap = -log(1 - random())
target = current_hazard + gap
next_idx = upper_bound(hazards, target)
// Cost: O(k_fired * log N_sites)
// Typical: k_fired ~ 5 out of 5000

Speedup: 100-1000x for realistic QEC circuits. The cumulative hazard table is precomputed by the compiler into noise_hazards[]. Binary search (std::upper_bound) finds the next firing site in O(log N).

void draw_next_noise(const vector<double>& hazards) {
    double current = (next_noise_idx == 0) ? 0.0 : hazards[next_noise_idx - 1];
    double gap     = -std::log(1.0 - random_double());   // Exponential(1)
    double target  = current + gap;
    auto it = std::upper_bound(hazards.begin(), hazards.end(), target);
    next_noise_idx = std::distance(hazards.begin(), it);  // may == size() => done
}

OP_NOISE_BLOCK: Batched Noise Processing

The noise_block_pass optimization groups consecutive noise sites [start, start+count) into a single instruction for a tight inner loop.

Without noise blocks

OP_NOISE  site_42   // dispatch overhead
OP_NOISE  site_43   // dispatch overhead
OP_NOISE  site_44   // dispatch overhead
OP_NOISE  site_45   // dispatch overhead
...                  // 100 more

With OP_NOISE_BLOCK

OP_NOISE_BLOCK  start=42, count=104
// Single dispatch, tight while loop:
while (next_noise_idx >= start
    && next_noise_idx < start+count) {
  exec_noise(state, pool, next_noise_idx);
}

Key insight: Most shots skip the entire block when next_noise_idx falls outside [start, start+count). The while-loop condition fails immediately -- one comparison instead of 100+ dispatch table lookups.

Shot Reset and Gamma Renormalization

reset() Between Shots

Zero the active region, restore initial state -- no reallocation.

void SchrodingerState::reset() {
  // Zero only active entries (not full array)
  for (i = 0; i < 2^active_k; i++)
    v_[i] = {0, 0};
  v_[0] = {1.0, 0.0};     // |0...0>

  fill(p_x, 0);            // clear frame
  fill(p_z, 0);
  gamma_ = {1.0, 0.0};
  active_k = 0;
  fill(obs_record, 0);

  // PRNG NOT reseeded -- streams forward
}

Gamma Renormalization

When |gamma| drifts outside [1e-100, 1e100], fold the magnitude into v_[].

void scale_magnitude(double scale) {
  gamma_ *= scale;
  double g_mag = std::abs(gamma_);

  if (g_mag > 1e100 ||
      (g_mag < 1e-100 && g_mag > 0.0)) {
    // Absorb magnitude into v[]
    for (i = 0; i < v_size(); i++)
      v_[i] *= g_mag;
    gamma_ /= g_mag;
    // gamma now ~unit magnitude
    // v[] carries the scale
  }
}

Uses std::abs() (not std::norm()) to avoid squaring-induced underflow: norm() squares the magnitude, so values near 1e-154 underflow to 0.

Summary

Why the SVM design works

The SVM is Elegant by Design

Compiler Does the Hard Work

Clifford absorption, Pauli localization, axis fusion, squeeze, noise blocking -- all happen at compile time. The VM sees only pre-optimized bytecode.

VM is Simple

A flat dispatch loop over a ~30-opcode ISA. Each kernel is a tight inner loop over v_[0..2^k]. No graph traversal, no dynamic scheduling.

Same Machine, Two Targets

CPU: SIMD within each shot (AVX-512 over the amplitude array). GPU: SIMT across shots (one thread per shot). The abstract machine is identical.

Performance Summary

Gap sampling: O(k_fired * log N) instead of O(N_sites). For p=0.001 and N=5000, we skip ~99.9% of noise sites.

ISA dispatch: AVX-512 processes 4 complex doubles (64 bytes) per cycle. Same loop runs scalar on ARM or older x86.

Memory reuse: Allocate once, zero only the active region between shots. Huge pages reduce TLB misses for large arrays.

Portable abstraction: The same bytecode runs on CPU (SIMD within shot) and GPU (SIMT across shots). The factored-state math is target-independent.

Bottom line: The SVM is a thin, fast interpreter for a well-optimized bytecode. The compiler does the thinking; the VM does the arithmetic.
