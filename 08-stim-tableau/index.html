What is Stim?

Google's fast stabilizer circuit simulator

Stim: Fast Clifford Simulation

Core Facts

  • Created by Craig Gidney at Google
  • Open source: github.com/quantumlib/Stim
  • Written in C++ with SIMD optimizations
  • Python bindings for scripting
  • Industry standard for QEC research

Gottesman-Knill Theorem

Any circuit composed purely of Clifford gates, preparations in |0⟩, and computational-basis measurements can be simulated in O(n²) time per gate on a classical computer.

Key Capabilities

  • Tableau simulator — full stabilizer state tracking
  • Frame simulator — fast Pauli frame sampling
  • Circuit format — .stim text files
  • Noise models — depolarizing, Pauli channels, readout
  • QEC support — detectors, observables, repeat blocks

Stim's Limitation → Clifft's Extension

Stim can only simulate purely Clifford circuits

↓

Real QEC circuits need non-Clifford gates (T gates, rotations)

↓

Clifft extends Stim by adding non-Clifford support

Stim (standalone)

  • Clifford gates only: H, S, CX, CZ, SWAP...
  • Measurements and noise
  • Cannot handle T, T†, R_Z(θ)

Clifft (built on Stim)

  • All Clifford gates (via Stim)
  • T, T†, R_X, R_Y, R_Z, U3
  • Compilation to optimized bytecode
  • GPU backends (CUDA, AMD)

What is the Stabilizer Tableau?

The mathematical engine behind Clifford gate absorption

Stabilizer States

Stabilizer Group

  • A quantum state |ψ⟩ is stabilized by Pauli operator P if P|ψ⟩ = |ψ⟩
  • The set of all such P forms the stabilizer group
  • For n qubits: the group has n generators
  • Each generator is an n-qubit Pauli string (tensor product of I, X, Y, Z)

Key Insight

A stabilizer state on n qubits is uniquely defined by its n stabilizer generators.

Instead of storing 2n complex amplitudes, we store n Pauli strings.

Example: |000⟩

Stabilizers: +Z1, +Z2, +Z3

Each Z stabilizes the corresponding |0⟩ eigenstate

The Tableau as a Binary Matrix

Each Pauli string is encoded as n X-bits + n Z-bits + 1 sign bit

Encoding: x=0, z=0 → I x=1, z=0 → X x=0, z=1 → Z x=1, z=1 → Y

Stim's Tableau Structure

template <size_t W>
struct Tableau {
    size_t num_qubits;
    TableauHalf<W> xs;  // X-output half
    TableauHalf<W> zs;  // Z-output half
};

struct TableauHalf<W> {
    size_t num_qubits;
    simd_bit_table<W> xt;    // X-bits
    simd_bit_table<W> zt;    // Z-bits
    simd_bits<W> signs;      // sign vector
};

Two Halves

  • xs: where each input Xi maps to (n rows)
  • zs: where each input Zi maps to (n rows)

The tableau tells us: "what does Xi (or Zi) conjugate to under the Clifford unitary?"

T(P) = U P U†

The Full Tableau: Stabilizers + Destabilizers

Two Halves, 2n Rows Total

Stim's Tableau stores:

  • xs half — n rows for X-output generators
  • zs half — n rows for Z-output generators

The xs half plays the role of destabilizers and the zs half plays the role of stabilizers (for the |0⟩⊗n convention).

Why Destabilizers?

Needed for measurement simulation. When measuring, we need to find generators that anti-commute with the measurement operator. Destabilizers complete the algebraic structure.

How the Tableau Simulates Gates

Each Clifford gate is a row operation on the binary matrix

Clifford Gate Update Rules

Every Clifford gate is a simple bitwise operation on the tableau rows

Gate Rule (applied to ALL rows) Stim method
H(i) Swap X[i] ↔ Z[i], update sign: s ^= X[i] & Z[i] prepend_H_XZ(q)
S(i) Z[i] ^= X[i], update sign: s ^= X[i] & Z[i] prepend_SQRT_Z(q)
CX(c,t) X[t] ^= X[c]; Z[c] ^= Z[t] prepend_ZCX(c,t)
CZ(a,b) Z[b] ^= X[a]; Z[a] ^= X[b] prepend_ZCZ(a,b)
SWAP(a,b) Swap columns a ↔ b in both X and Z prepend_SWAP(a,b)
X(i) s ^= Z[i] (flip sign where Z-bit is set) prepend_X(q)
Z(i) s ^= X[i] (flip sign where X-bit is set) prepend_Z(q)

All operations are O(n) per gate — iterate over n rows, apply bitwise XOR per row. With SIMD: O(n/W) where W is SIMD width.

Step-Through: H(0) → CX(0,1) → M(0,1)

Step 0: Initial state |00⟩

Tableau = identity. X0→X0, X1→X1, Z0→Z0, Z1→Z1

Step 1: Apply H(0)

Swap X↔Z for qubit 0. Now: X0→Z0, Z0→X0

State is now |+0⟩ = (|00⟩ + |10⟩)/√2

Step 2: Apply CX(0,1)

X[1] ^= X[0], Z[0] ^= Z[1]. Now: X0→Z0Z1, X1→X0X1

State is now Bell state (|00⟩ + |11⟩)/√2

Step 3: Measure qubit 0 (Z basis)

Check: does any stabilizer anti-commute with Z0? Yes — X0→Z0Z1 anti-commutes. Random outcome, collapse the tableau.

Step 4: Measure qubit 1 (Z basis)

After collapse, Z1 is determined. Deterministic outcome (same as qubit 0's result).

Measurement in the Tableau

Algorithm

  1. Given measurement observable P (e.g., Zi)
  2. Scan stabilizer rows: does any row anti-commute with P?
  3. If YES (random case):
    • Flip a coin for the outcome
    • Collapse: replace the anti-commuting row
    • Row-reduce to restore canonical form
  4. If NO (deterministic case):
    • The outcome is determined by the current state
    • Compute from the sign of the stabilizer

Anti-commutation Check

Two Pauli strings P, Q anti-commute iff their symplectic inner product is 1:

P = (x_P, z_P)
Q = (x_Q, z_Q)

[P, Q] = x_P · z_Q + z_P · x_Q (mod 2)

This is a single bitwise AND + popcount + XOR — extremely fast with SIMD

Noise as Pauli Kick

Stochastic noise (e.g., X_ERROR(p)) is just a random XOR on the tableau sign bits or X/Z bits. Same complexity as a gate update.

SIMD-Accelerated Tableau Operations

Stim uses simd_bits and simd_bit_table for all tableau storage

How It Works

  • Each row (Pauli string) stored as SIMD-aligned bit arrays
  • Gate updates = vectorized XOR/AND over entire rows
  • A single H gate on qubit i touches ALL 2n rows
  • With 256-bit AVX2: process 256 bits per instruction
  • 1000-qubit tableau row: ~16 SIMD XOR ops instead of 1000 scalar ops

Column-Major Layout

Stim's memory layout is column-major:

  • Iterating over output qubits is along the grain of memory
  • Makes prepending operations cheap
  • For appending, uses TableauTransposedRaii (temporary transpose)

Clifft's backend uses TableauTransposedRaii for lazy flush of pending gates in the VirtualFrame

Parser Decompositions

Complex gates are decomposed at parse time:

  • CH(c,t) → R_Y(0.25) t, CX c t, R_Y(-0.25) t
  • CCZ(a,b,c) → 7 T/T† + 6 CX
  • CCX(a,b,t) → H t, CCZ a b t, H t

The parser's gate table has ~80 gate types — see gate_data.h

is_clifford(gate) is a constexpr O(1) lookup into kGateTraitsData[] — the frontend uses this to decide: absorb or emit?

Stim's Two Simulation Modes

1. Tableau Simulator

  • Tracks the full stabilizer state
  • Maintains complete 2n × 2n tableau
  • Can answer: "what is the state?"
  • Used for: debugging, state inspection
  • Cost: O(n²) per gate

This is what Clifft's frontend uses!

2. Frame Simulator

  • Tracks only the Pauli frame
  • 2 × n bits per sample (p_x, p_z)
  • Cannot answer: "what is the state?"
  • Used for: fast sampling of detection events
  • Cost: O(n) per gate

Conceptually similar to Clifft's SVM Pauli frame at runtime

Clifft uses the tableau simulator at compile time and a frame-like approach at runtime — best of both worlds

Frontend trace() — The Core Algorithm

From frontend/frontend.h:

// 1. Initialize TableauSimulator with identity tableau
// 2. For each gate in the circuit:
//    - Clifford: apply to simulator (absorbed into tableau)
//    - T/T_DAG: extract rewound Z from inv_state.zs[q], emit HeisenbergOp
//    - Measurement: extract rewound observable, emit MEASURE
//    - Classical feedback: extract rewound Pauli, emit CONDITIONAL_PAULI
// 3. Return HirModule with all emitted operations

Clifford Gates: Absorbed

  • Gate applied to the Stim Tableau
  • No HIR op emitted
  • Zero cost at runtime
  • e.g., H, S, CX, CZ, SWAP, X, Y, Z

Non-Clifford Gates: Emitted

  • Tableau used to extract rewound Pauli
  • HeisenbergOp emitted with Pauli mask
  • The mask encodes which qubits are affected
  • e.g., T, T†, R_Z(θ)

The gate classification is O(1): is_clifford(gate) indexes into kGateTraitsData[]

The Heisenberg Picture Transformation

Schrödinger Picture

Transform the state forward:

|ψ'⟩ = U |ψ⟩

Apply gate matrices to state vector. Cost: O(2n)

Heisenberg Picture

Transform the observables backward:

P' = U† P U

Conjugate Pauli operators through the inverse. Cost: O(n) for Clifford

How the Frontend Uses This

Gate sequence:  H(0)  CX(0,1)  T(0)  CZ(0,2)  M(1)

Tableau:     [absorb] [absorb]  |    [absorb]   |
                                  |                 |
HIR ops:                     [T_GATE         [MEASURE
                           mask from        mask from
                           tableau]         tableau]

Clifford gates update the tableau silently.
Non-Clifford gates query the tableau for their Pauli mask, then emit.

The Rewound Pauli: Concrete Example

Circuit: H(0), CX(0,1), T(0)

After absorbing H(0) and CX(0,1), the tableau says:

  • Z0 → X0 (via the inverse)
  • More precisely: inv_state.zs[0] = X0X1

When T(0) is encountered:

  • Frontend reads inv_state.zs[0]
  • Gets the multi-qubit Pauli string X0X1
  • This becomes the mask of the T_GATE HeisenbergOp

Why "Rewound"?

The mask tells us: "In the virtual (Heisenberg) basis, this T gate's Z observable has been spread by the preceding Cliffords into a multi-qubit Pauli string."

The HIR op stores: PauliMaskHandle → indexes into HirModule::pauli_masks (a PauliMaskArena)

Arena Storage (from hir.h)

hir.append_tgate(/*dagger=*/false, [&](MutablePauliMaskView slot) {
    stim_to_mask_view(rewound.xs, n, slot.x());  // X-bits of rewound Pauli
    stim_to_mask_view(rewound.zs, n, slot.z());  // Z-bits of rewound Pauli
    slot.set_sign(rewound.sign);                  // sign bit
});

What Survives into the Compiled Bytecode

NOT in the Bytecode

  • The Stim Tableau itself
  • Any Clifford gate operations
  • Tableau algebra or conjugation

The tableau's effect is fully captured during compilation

IN the Bytecode

  • OP_FRAME_* ops: bit flips on p_x/p_z
  • OP_ARRAY_T/T_DAG: with precomputed axis
  • OP_MEAS_*: localized measurement
  • OP_APPLY_PAULI: precomputed N-bit masks
  • ConstantPool: masks, noise, fused unitaries

The Journey of a Clifford Gate

CX(0,1) in .stim
    ↓
AstNode{GateType::CX, targets=[0,1]} in Circuit AST
    ↓
Absorbed into Stim Tableau in Frontend  ← no HIR op emitted!
    ↓
Effect captured in Pauli masks of neighboring T/M ops
    ↓
OP_FRAME_CNOT or merged into OP_ARRAY ops in bytecode

Compiled vs Interpreted

Why Clifft compiles instead of interpreting like Stim

Stim: Interpreted — Clifft: Compiled

Stim (Standalone)

INTERPRETED

  • Walks circuit instruction by instruction
  • Each gate: dispatch + tableau update
  • Compilation cost: zero
  • Per-gate cost: higher
  • No optimization across gates

Clifft

COMPILED

  • Multi-stage pipeline: Parse → Trace → Optimize → Lower → Optimize
  • Upfront cost: compilation time
  • Per-shot cost: much lower
  • 11 optimization passes
  • Gate fusion, squeeze, skip-ahead

Key Differences

Aspect Stim (Interpreted) Clifft (Compiled)
Instruction level High-level gates (H, CX...) Low-level bytecode (OP_FRAME_H, OP_ARRAY_T...)
Clifford gate cost O(n) per gate per shot O(1) bit flip per shot (frame ops)
Non-Clifford Not supported T, rotations, fused U2/U4
Optimization None across gates Peephole, squeeze, fusion, multi-gate
Noise sampling Per-site RNG draw Skip-ahead exponential gap
Instruction size Variable (gate + targets) Fixed 32 bytes (cache-aligned)
GPU backends No CUDA + HIP

When Compilation Wins

Stim (interpret per shot):
Shot 1: [parse][gate][gate][gate][...][gate]
Shot 2: [parse][gate][gate][gate][...][gate]
Shot 3: [parse][gate][gate][gate][...][gate]
  ...     ...   same work repeated every shot

Clifft (compile once, execute many):
Compile: [parse][trace][optimize][lower][optimize]  (one-time cost)
Shot 1:  [exec bytecode]  (fast: no tableau, no dispatch overhead)
Shot 2:  [exec bytecode]
Shot 3:  [exec bytecode]
  ...      amortized over billions of shots

Compilation Wins When:

  • Many shots of the same circuit (amortize compilation over 106–1012 shots)
  • Optimization passes reduce per-shot work (StatevectorSqueeze, fusion passes)
  • GPU execution needs flat bytecode (no pointer chasing)

The Bytecode is Pre-Digested:

  • All Clifford gates absorbed (zero runtime cost)
  • Non-Clifford ops have precomputed Pauli masks
  • Gate sequences fused into U2, U4
  • Noise sites indexed for skip-ahead sampling

The Pauli Frame

Runtime successor of the compile-time tableau

Compile Time: Tableau — Runtime: Pauli Frame

Compile Time: Full Tableau

  • 2n × 2n binary matrix + sign vector
  • Tracks the accumulated Clifford unitary UC
  • Used to extract Pauli masks for non-Clifford ops
  • Memory: O(n²) bits

Runtime: Pauli Frame

  • Two n-bit arrays: p_x[] and p_z[]
  • Tracks stochastic bit-flips and phase-flips
  • Updated by OP_FRAME_* ops via XOR
  • Memory: O(n) bits

Why the Frame is Enough

After compilation, every operation in the bytecode is either:

  • Frame operation — updates p_x/p_z bits directly (OP_FRAME_CNOT, OP_FRAME_H, ...)
  • Array operation — operates on the amplitude vector v[] with precomputed axis targets

The tableau did its job during compilation — it extracted all the masks and transformations. No tableau math needed at runtime.

From Tableau to Pauli Frame

COMPILE TIME: Full Tableau (O(n²) bits)

     X-half (xs)              Z-half (zs)
  q0 q1 q2 q3 ...         q0 q1 q2 q3 ...        sign
  [1  0  1  0 ...]         [0  1  0  1 ...]        [+]     row 0
  [0  1  0  0 ...]         [1  0  0  1 ...]        [-]     row 1
  [1  1  1  0 ...]         [0  0  1  0 ...]        [+]     row 2
  [0  0  0  1 ...]         [0  1  1  1 ...]        [+]     row 3
     ...                       ...                  ...    2n rows

     ↓↓↓ compilation ↓↓↓

RUNTIME: Pauli Frame (O(n) bits)

  p_x: [0  1  0  0  ...]     n bits total
  p_z: [1  0  0  1  ...]     n bits total

  Updated per-shot by OP_FRAME_* instructions via XOR

Compression ratio: The compilation step compresses O(n²) bits of tableau into O(n) bits of frame + precomputed bytecode masks

For 1000 qubits: tableau ~125 KB, frame ~250 bytes (per shot)

Frame Opcodes in the Bytecode

Descendants of tableau operations, but O(1) per operation at runtime

OpcodeFrame Effect
OP_FRAME_HSwap p_x[v] ↔ p_z[v]
OP_FRAME_Sp_z[v] ^= p_x[v]
OP_FRAME_CNOTp_x[t] ^= p_x[c]; p_z[c] ^= p_z[t]
OP_FRAME_CZp_z[a] ^= p_x[b]; p_z[b] ^= p_x[a]
OP_FRAME_SWAPSwap p_x[a]↔p_x[b], p_z[a]↔p_z[b]

Compare: Tableau vs Frame

TableauFrame
H on qubit iO(n) row ops1 bit swap
CNOT(c,t)O(n) row ops2 bit XORs
CZ(a,b)O(n) row ops2 bit XORs

Frame ops touch individual bits. Tableau ops touch entire rows of O(n) bits. That is the payoff of compilation.

inline void stim_to_mask_view(const stim::simd_bits_range_ref<kStimWidth>& bits,
                              uint32_t n, MutableMaskView dst) {
    const uint32_t words = (n + 63) / 64;
    for (uint32_t w = 0; w < words; ++w)

Summary

The tableau is the mathematical engine of Clifft's compilation

Key Takeaways

The Tableau's Role

  1. The tableau is the mathematical engine that makes Clifft's compilation work
  2. It absorbs ALL Clifford gates (which dominate QEC circuits: 99%+ are Clifford) into a compact binary matrix
  3. Non-Clifford gates are "extracted" relative to the tableau as Pauli masks via the Heisenberg picture
  4. At runtime, the lightweight Pauli frame replaces the tableau

Stim + Clifft: A Partnership

STIM provides the FOUNDATION:
  • .stim circuit format (de facto QEC standard)
  • Gate catalog and classification
  • Tableau algebra (SIMD-accelerated)
  • PauliString type and operations

CLIFFT extends with:
  • Non-Clifford gate support (T, rotations)
  • AOT compilation pipeline (5 stages)
  • 11 optimization passes (peephole, squeeze, fusion)
  • Bytecode VM with SIMD dispatch (AVX2, AVX-512)
  • GPU backends (CUDA, HIP/ROCm)
  • Factored state representation (|ψ⟩ = γ U_C P (|φ⟩ ⊗ |0⟩))

The tableau converts an n-qubit quantum problem into a k-qubit quantum problem (k << n)

Where k = peak active dimension = number of non-Clifford contaminated qubits

Cost: O(2n) → O(2k)
