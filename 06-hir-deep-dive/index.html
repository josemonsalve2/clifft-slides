Heisenberg Intermediate Representation

Stim Text → Parser → Circuit AST → Frontend → HIR → Optimizer → Backend → Bytecode

The central data structure that makes near-Clifford simulation tractable

Section 1

What is the HIR?

Position in the Pipeline

Stim Text → parse() → Circuit AST → trace() → HirModule
HirModule → HIR Passes → lower() → CompiledModule → SVM

Why an Intermediate Representation?

  • Separation of concerns -- decouple parsing from optimization from code generation
  • The Circuit AST is a faithful representation of the input text (gate-by-gate)
  • The HIR is a semantic representation: Clifford gates are absorbed, only non-Clifford ops + measurements remain
  • The Bytecode is a machine-level encoding with localized single-axis operations
  • Each stage can evolve independently -- new Stim gates don't affect the backend, new optimizations don't affect parsing

The Heisenberg Picture

Schrodinger Picture

Track how the state evolves forward through gates

|psi> = U_n ... U_2 U_1 |0>

// Each gate transforms the state vector
// Cost: O(2^n) per gate

This is what the SVM does at runtime -- it carries an amplitude vector v[2^k]

Heisenberg Picture

Track how observables evolve backward through gates

O' = U_1^dag ... U_n^dag O U_n ... U_1

// Clifford gates map Paulis to Paulis
// Cost: O(n^2/64) per Clifford gate

This is what the Frontend does at compile time -- it tracks a Stim Tableau

The key insight: Clifford gates are "free" in the Heisenberg picture. They just update the tableau mapping. Only non-Clifford gates (T, rotations) and measurements produce HIR operations.

What Does the HIR Actually Contain?

IN the HIR (emitted as HeisenbergOps)

  • T gates -- with "rewound" Pauli masks showing which qubits are affected
  • Measurements -- with rewound observable Pauli masks
  • Classical feedback -- conditioned Pauli corrections
  • Noise channels -- stochastic Pauli errors
  • Phase rotations -- continuous Z-rotations (generalized T)
  • Detectors / Observables -- QEC parity checks
  • Expectation values -- non-destructive probes

NOT in the HIR (absorbed into tableau)

  • H (Hadamard)
  • S, S_DAG (Phase gates)
  • X, Y, Z (Pauli gates)
  • CX/CNOT (Controlled-X)
  • CZ (Controlled-Z)
  • SWAP
  • ISWAP, SQRT_XX, SQRT_YY, SQRT_ZZ
  • All 23 single-qubit Cliffords
  • All 22 two-qubit Cliffords

45 Clifford gate types produce zero HIR output

Heisenberg vs Schrodinger -- Visually

Schrodinger: State Evolves Forward

|0> —— [H] —— [CNOT] —— [T] —— [M] —> result
|0> —————— [CNOT] ——————— [M] —> result
State transforms: |00> → |+0> → |phi+> → T|phi+> → collapse

Heisenberg: Observables Evolve Backward

rewind <— [H] <— [CNOT] <— [T] <— [M]
rewind <————— [CNOT] <—————— [M]
Z_0 rewound through Cliffords: Z_0 → X_0 → X_0⊗X_1

The "rewound Pauli": When the frontend encounters T 0, it asks "what does Z_0 look like after undoing all Cliffords so far?" The answer is a multi-qubit Pauli string -- this becomes the T gate's Pauli mask in the HIR.

Mathematically: inv_state.zs[q] gives the Heisenberg-picture Z observable for qubit q conjugated through the inverse of accumulated Cliffords.

Section 2

The Frontend -- Circuit to HIR

The trace() Function

// frontend.h -- Main entry point
[[nodiscard]] HirModule trace(const Circuit& circuit);

Algorithm Overview

Initialize a Stim TableauSimulator with an identity tableau (n qubits)
Pre-count Pauli masks + noise channels to pre-size PauliMaskArena storage
Walk each AstNode in the circuit sequentially
For each gate, dispatch on its type:
  • Clifford → apply to simulator (absorbed into tableau, no HIR output)
  • T / T_DAG → extract rewound Pauli, emit HeisenbergOp::T_GATE
  • Measurement → extract rewound observable, emit HeisenbergOp::MEASURE
  • Feedback → extract rewound Pauli, emit HeisenbergOp::CONDITIONAL_PAULI
  • Noise → build noise site, emit HeisenbergOp::NOISE
  • R_X/R_Y/R_Z/U3 → decompose into Clifford + rotation, emit PHASE_ROTATION
Return the assembled HirModule with ops, arenas, side-tables, and final tableau

Clifford Gates: Absorbed into Tableau

When the frontend encounters a Clifford gate, it calls the corresponding Stim simulator method:

Stim GateSimulator Method
Hdo_H_XZ(q)
Sdo_S(q)
CXdo_ZCX(c, t)
CZdo_ZCZ(a, b)
SWAPdo_SWAP(a, b)
X / Y / Zdo_X/Y/Z(q)
ISWAPdo_ISWAP(a, b)

What "absorption" means

The tableau tracks how each Pauli Z_q and X_q transforms under the accumulated Clifford unitary.

After H 0; CNOT 0 1:

Z_0 maps to X_0 * X_1  (= X tensor X)
X_0 maps to Z_0 * Z_1  (= Z tensor Z)
Z_1 maps to Z_1
X_1 maps to X_0 * X_1

This is stored as rows of a 2n x 2n binary matrix in Stim's tableau.

Result: Clifford gates produce zero HIR ops. A pure-Clifford circuit with 10,000 gates produces an empty ops vector. Only the final tableau survives.

T Gates: Rewound Through Tableau

When the frontend encounters T q (a pi/8 phase on qubit q):

Extract the "rewound Z" from the tableau: inv_state.zs[q]
This is the n-qubit Pauli string that Z_q maps to under the inverse of accumulated Cliffords
Claim a fresh slot in HirModule::pauli_masks arena
Fill the slot with the rewound Pauli's X-bits, Z-bits, and sign: 
hir.append_tgate(/*dagger=*/false, [&](MutablePauliMaskView slot) {
    stim_to_mask_view(rewound.xs, n, slot.x());
    stim_to_mask_view(rewound.zs, n, slot.z());
    slot.set_sign(rewound.sign);
});

Why rewinding matters

Consider: H 0; T 0. Without H, T acts as diag(1, e^{i*pi/4}) on Z_0. But with the preceding H, the "rewound" Z is actually X_0. The HIR records the T gate as acting on the Pauli X_0 -- the backend then knows to expand qubit 0 in the X basis.

Measurements and Classical Feedback

Measurements

For M q (Z-basis measurement on qubit q):

  • Extract rewound observable: inv_state.zs[q]
  • Emit HeisenbergOp::MEASURE with the Pauli mask and a MeasRecordIdx
  • For MX q: extract inv_state.xs[q] instead
  • For MPP: combine Pauli-tagged targets into a product mask

The rewound mask tells the backend exactly which qubits are involved and in which basis

Classical Feedback (CONDITIONAL_PAULI)

For CX rec[-k] q ("apply X to q if measurement k was 1"):

  • The controlling measurement is rec[-k]
  • The applied Pauli X_q is rewound through the tableau
  • Emit HeisenbergOp::CONDITIONAL_PAULI with:
    • ControllingMeasIdx: which measurement result controls this
    • Pauli mask: the rewound correction to apply

Used in QEC for Pauli frame corrections after syndrome measurement

Worked Trace: H 0; CNOT 0 1; T 2; M 0 1 2

Gate Action Tableau Change HIR Output
H 0 Absorbed Z_0 ↔ X_0 (none)
CNOT 0 1 Absorbed Z_0 → X_0⊗X_1
Z_1 → Z_0⊗Z_1		 (none)

M 0 Rewound & emitted inv_state.zs[0] = X_0⊗X_1 MEASURE mask: X⊗X⊗I

Result: 4 HIR ops (1 T_GATE + 3 MEASURE), 2 Clifford gates fully absorbed. The T gate's mask is I⊗I⊗Z because qubit 2 wasn't touched by any Clifford.

REPEAT Block Handling

REPEAT 100 {
  H 0
  CNOT 0 1
  M 0 1
  DETECTOR rec[-1] rec[-3]
}

REPEAT blocks are unrolled at parse time via text-level replay. The body is re-parsed N times against the same Circuit.

This means rec[-k] references naturally resolve to correct absolute measurement indices as the repeat count grows.

Safety limit: kMaxUnrolledOps = 10,000,000 total AST nodes to prevent OOM.

HIR Data Structures -- In Detail

HeisenbergOp: 16-Byte Layout

struct HeisenbergOp {                // Exactly 16 bytes
    PauliMaskHandle mask_handle_;    // 4 bytes  (offset 0)
    OpType type_;                    // 1 byte   (offset 4)
    uint8_t flags_;                  // 1 byte   (offset 5)
    uint8_t pad_[2];                 // 2 bytes  (offset 6)
    union { /* payload */ };         // 8 bytes  (offset 8)
};
static_assert(sizeof(HeisenbergOp) == 16);

Memory layout (4 ops per 64-byte L1 cache line):

Byte offset 0123 4567 891011 12131415
Field mask_handle (u32) type flags pad[2] union payload (8 bytes, double-aligned)

mask_handle_: index into HirModule::pauli_masks arena. Value kNoMask = 0xFFFFFFFF for ops without Pauli masks (NOISE, DETECTOR, OBSERVABLE, READOUT_NOISE).

flags_: FLAG_IS_DAGGER (0x01) for T_DAG vs T; FLAG_HIDDEN (0x04) for hidden measurements (from resets).

Payload Union Details

// Per-OpType payload variants (8 bytes each, at offset 8)
union {
    MeasurePayload measure_;         // { uint32_t meas_record_idx; }
    ConditionalPayload conditional_; // { uint32_t controlling_meas; }
    NoisePayload noise_;             // { uint32_t site_idx; }
    ReadoutPayload readout_;         // { uint32_t entry_idx; }
    DetectorPayload detector_;       // { uint32_t target_list_idx; }
    ObservablePayload observable_;   // { uint32_t obs_idx;
                                     //   uint32_t target_list_idx; }
    PhasePayload phase_;             // { double alpha; }   -- full 8 bytes!
    ExpValPayload exp_val_;          // { uint32_t exp_val_idx; }
};

4-byte payloads

Most payloads use only 4 bytes (a single uint32_t index). The remaining 4 bytes of the 8-byte union are padding.

Exception: ObservablePayload uses both uint32_t fields (obs_idx + target_list_idx = 8 bytes).

8-byte payload

PhasePayload stores a full double alpha -- the rotation angle in half-turns. This is the only variant that uses the full 8 bytes as a floating-point value.

The union is double-aligned (offset 8) to ensure alpha is naturally aligned for fast load/store.

Lambda-fill builders on HirModule

Ops are NOT constructed directly. Instead, HirModule::append_* builders atomically: claim an arena slot → invoke a fill callable → append the op.

// Example: appending a T gate
hir.append_tgate(/*dagger=*/false, [&](MutablePauliMaskView slot) {
    stim_to_mask_view(rewound.xs, n, slot.x());  // write X-bits
    stim_to_mask_view(rewound.zs, n, slot.z());  // write Z-bits
    slot.set_sign(rewound.sign);                  // write sign
});

This pattern guarantees that arena allocation + mask fill + op push_back happen atomically at each call site.

Strong Typedefs: Type-Safe Indices

// Strong typedefs prevent mixing up different index types
enum class MeasRecordIdx : uint32_t {};       // Absolute measurement position
enum class ControllingMeasIdx : uint32_t {};  // Which measurement controls a CONDITIONAL_PAULI
enum class NoiseSiteIdx : uint32_t {};        // Index into noise_sites[]
enum class DetectorIdx : uint32_t {};         // Index into detector_targets[]
enum class ObservableIdx : uint32_t {};       // Index into observable_targets[]
enum class ExpValIdx : uint32_t {};           // Index into exp_val records
enum class ReadoutNoiseIdx : uint32_t {};     // Index into readout_noise[]
enum class PauliMaskHandle : uint32_t {};     // Index into PauliMaskArena

Why strong typedefs? Passing a NoiseSiteIdx where a DetectorIdx is expected is a compile-time error. All are uint32_t underneath, but the type system prevents accidental misuse.

The kNoMask sentinel (PauliMaskHandle(~0u)) marks ops that carry no Pauli mask. Checked via op.has_mask().

Determinism Check

[[nodiscard]] bool is_deterministic() const {
    for (const auto& op : ops) {
        if (type == OpType::MEASURE || type == OpType::NOISE ||
            type == OpType::READOUT_NOISE || type == OpType::CONDITIONAL_PAULI)
            return false;
    }
    return true;  // Pure unitary: well-defined global phase
}

Deterministic modules have a well-defined statevector including global phase. Stochastic ones (with measurements/noise) are only defined per shot.

Section 4

The Pauli Mask System

Pauli Encoding Convention

Each qubit's Pauli operator is encoded using two bits: one from the X-array, one from the Z-array.

I

X=0, Z=0

X

X=1, Z=0

Z

X=0, Z=1

Y

X=1, Z=1

5-Qubit Example: X ⊗ I ⊗ Z ⊗ Y ⊗ I

Qubit q0 q1 q2 q3 q4
X bits 1 0 0 1 0
Z bits 0 0 1 1 0
Pauli X I Z Y I

Plus a sign bit: 0 = positive (+XIZYI), 1 = negative (-XIZYI). Stored separately in signs_[].

PauliMaskArena: Contiguous Storage

class PauliMaskArena {
    uint32_t num_words_;          // ceil(num_qubits / 64) -- fixed at construction
    std::vector<uint64_t> x_;    // All X-bits, contiguous  [mask0_x | mask1_x | ...]
    std::vector<uint64_t> z_;    // All Z-bits, contiguous  [mask0_z | mask1_z | ...]
    std::vector<uint8_t> signs_; // One sign byte per mask  [s0 | s1 | ...]

    PauliMaskView at(PauliMaskHandle h) const;        // Read-only view
    MutablePauliMaskView mut_at(PauliMaskHandle h);   // Mutable view
};

Memory Layout (3 masks, 128 qubits = 2 words)

x_[]: m0_x_w0m0_x_w1m1_x_w0m1_x_w1m2_x_w0m2_x_w1
z_[]: m0_z_w0m0_z_w1m1_z_w0m1_z_w1m2_z_w0m2_z_w1
signs_[]: s0s1s2

Why parallel arrays? Keeping x_[] and z_[] in separate contiguous arrays optimizes for the common case of iterating only X or only Z bits (e.g., the backend's localization pass reads X support and Z support independently).

Fixed capacity: The arena is pre-sized at construction. No resizing means all PauliMaskView references (which hold raw span pointers) remain valid for the arena's lifetime.

Mask Views: Read-Only and Mutable

class PauliMaskView {
    MaskView x_;          // span<const uint64_t> over the X-bit words for one mask
    MaskView z_;          // span<const uint64_t> over the Z-bit words for one mask
    const uint8_t* sign_; // pointer to the sign byte

    MaskView x() const;   // Read X-bits
    MaskView z() const;   // Read Z-bits
    bool sign() const;    // Read sign
};

class MutablePauliMaskView {
    MutableMaskView x_;   // span<uint64_t> -- writable
    MutableMaskView z_;   // span<uint64_t> -- writable
    uint8_t* sign_;       // writable sign

    void set_sign(bool s);
    operator PauliMaskView() const;  // implicit conversion for read-only callers
};

HirModule Mask Accessors

// Convenience accessors on HirModule

The names "destab" and "stab" come from the stabilizer formalism: X-bits correspond to destabilizer rows, Z-bits to stabilizer rows in the tableau.

BitMask<N>: Compile-Time Fixed Width

template <size_t N>
struct BitMask {
    static constexpr size_t kWords = N / 64;
    std::array<uint64_t, kWords> w{};

    // Bitwise: operator^, &, |, ~, ^=, &=, |=
    // Queries: is_zero(), popcount(), lowest_bit()
    // Bit-level: bit_get(idx), bit_set(idx, val), bit_xor(idx), bit_swap(i, j)
    // Multi-word: clear_lowest_bit()  -- analog of x &= x - 1
};

N = 64 (1 word)

Single uint64_t. All operations compile to single machine instructions.

N = 512 (8 words = 64 bytes)

Fits exactly one AVX-512 register. XOR/AND/popcount auto-vectorize to single SIMD instructions at -O3.

Two-Tier Width System

BitMask<N> is for compile-time widths (auto-vectorization). MaskView wraps span<uint64_t> for runtime-width operations. Adapters view() / mut_view() bridge the two. The HIR uses MaskView since qubit count is determined at runtime.

HirModule: What It Owns

struct HirModule {
    // === Pauli mask storage ===
    PauliMaskArena pauli_masks;            // Gate masks (T, MEASURE, CONDITIONAL, etc.)

    uint32_t num_measurements = 0;

};

Internal State: CompilerContext

struct CompilerContext {

    std::vector<Instruction> bytecode;    // Output instruction stream

The CompilerContext is initialized with num_qubits and a flush threshold of 128. All axes start Dormant (k=0).

Pauli Localization: The Core Algorithm

[[nodiscard]] LocalizationResult localize_pauli(
    CompilerContext& ctx, const stim::PauliString<kStimWidth>& pauli);

struct LocalizationResult {
    uint16_t pivot;        // The virtual axis the Pauli was localized onto
    LocalizedBasis basis;  // Z_BASIS or X_BASIS
    bool sign;             // Accumulated sign (true = negative)
};
Count support: determine which qubits have non-identity Pauli factors (popcount of X | Z bits)
Greedy CNOT/CZ reduction: for each extra support qubit, emit a virtual CNOT or CZ to XOR its Pauli factor onto a chosen pivot qubit. Each virtual gate reduces support by 1.
Basis alignment: if the pivot has both X and Z set (i.e., Y), emit H or S to reduce to pure X or pure Z.
Result: the Pauli is now a single-qubit X_pivot or Z_pivot, ready for a single-axis VM operation.

All emitted virtual gates go into both the VirtualFrame (for future Pauli mappings) and the bytecode (as OP_FRAME_* or OP_ARRAY_* depending on whether the axis is active).

VirtualFrame: Lazy Clifford Composition

class VirtualFrame {
    stim::Tableau<kStimWidth> materialized_;  // Fully composed tableau
    std::vector<PendingGate> pending_gates_;  // Queue of un-flushed gates
    std::vector<PendingGate> gate_log_;       // Every gate ever appended
    size_t flush_threshold_;                   // Default: 128
};

map_pauli() -- Lazy Path

  • Called once per mask-carrying HIR op (T_GATE, MEASURE, etc.)
  • If queue length < threshold: replay pending gates inline over the Pauli string
  • This avoids the O(n^2) cost of flushing the full tableau
  • Empirically, the queue between consecutive HIR ops is short (just the gates from the previous localize_pauli)

Eager flushing here regresses T-heavy circuits (QV-20) by ~40%.

map_noise_channel() -- Eager Flush

  • Called once per noise channel (e.g., DEPOLARIZE2 has 15 channels)
  • Flushes the pending queue eagerly before mapping
  • Amortizes one O(n^2) flush over many channel mappings
  • QEC circuits have thousands of noise sites

PendingGateType: CNOT, CZ, S, H, SWAP. Each gate is replayed over a (MutableMaskView x, MutableMaskView z, bool& sign) triple using explicit bit manipulation (not Stim calls).

Active / Dormant Partition

class VirtualRegisterManager {
    uint32_t num_qubits_;   // Total n
    uint32_t active_k_;     // Current active count
    uint32_t peak_k_ = 0;  // Maximum k ever reached

    bool is_active(uint16_t axis)  const { return axis < active_k_; }
    bool is_dormant(uint16_t axis) const { return axis >= active_k_; }
    void activate();    // k -> k+1 (axis k becomes active)
    void deactivate();  // k -> k-1 (axis k-1 becomes dormant)
};
Axes 0 1 ... k-1 k k+1 ... n-1
← Active (touch v[2^k]) → ← Dormant (frame-only, O(1)) →
  • OP_EXPAND: activates axis k (k → k+1). Must SWAP pivot to boundary first.
  • Active measurement: deactivates highest axis (k → k-1).
  • peak_rank: maximum k ever reached. Determines the CompiledModule's array size: 2^peak_rank.
  • Safety check: validate_peak_rank(peak) rejects if peak ≥ 63 (undefined behavior in 1ULL << k shifts).

Noise Hazard Table and Fused Unitaries

Cumulative Hazard Table

// noise_hazards[i] = sum of
//   -log(1 - prob_sum_j)
// for noise sites j = 0..i

Enables O(1) skip of silent noise sites via exponential gap sampling. Instead of rolling the RNG for every noise site, the SVM draws from an exponential distribution and binary-searches the hazard table to find the next site that fires.

On QEC circuits with thousands of noise sites but low error rates, this can skip hundreds of silent sites per gap.

Fused Unitary Nodes

struct FusedU2Node {
  // For each of 4 Pauli frame states
  // ((p_z<<1)|p_x: 0=I, 1=X, 2=Z, 3=Y)
  complex<double> matrices[4][4];
  complex<double> gamma_multipliers[4];
  uint8_t out_states[4];
};

struct FusedU4Node {
  struct Entry {
    complex<double> matrix[4][4];
    complex<double> gamma_multiplier;
    uint8_t out_state;
  };
  Entry entries[16]; // 4-bit frame state
};

Created by the optimizer's SingleAxisFusion and TileAxisFusion passes.

Separation principle: 32-byte instructions stay small and cache-friendly. Variable-size data (Pauli masks spanning ceil(n/64) words, noise channels, U2/U4 matrices) lives in the constant pool, referenced by integer index. Two instructions fit per 64-byte cache line.

struct alignas(32) Instruction {       // Exactly 32 bytes
    Opcode opcode;      // 1 byte   (offset 0)

    uint16_t axis_2;    // 2 bytes  (offset 6) -- virtual axis target 2
    union { ... };      // 24 bytes (offset 8..31)
};

Opcode Categories at a Glance

Frame Opcodes

O(1) per qubit. Update only p_x/p_z.

  • OP_FRAME_CNOT
  • OP_FRAME_CZ
  • OP_FRAME_H
  • OP_FRAME_S / S_DAG
  • OP_FRAME_SWAP

Array Opcodes

O(2^k) sweep over v[].

  • OP_ARRAY_T / T_DAG
  • OP_ARRAY_CNOT / CZ
  • OP_ARRAY_H / S / S_DAG
  • OP_ARRAY_MULTI_CNOT
  • OP_ARRAY_ROT
  • OP_ARRAY_U2 / U4

Expansion

k → k+1, doubles array.

  • OP_EXPAND
  • OP_EXPAND_T / T_DAG
  • OP_EXPAND_ROT

Measurement

  • OP_MEAS_DORMANT_STATIC -- deterministic from p_x
  • OP_MEAS_DORMANT_RANDOM -- coin flip, phase to gamma
  • OP_MEAS_ACTIVE_DIAGONAL -- Z-basis filter, k→k-1
  • OP_MEAS_ACTIVE_INTERFERE -- X-basis fold, k→k-1
  • OP_SWAP_MEAS_INTERFERE -- fused swap + measure
  • Each has a _FORCED variant for importance sampling

Classical / Noise

  • OP_APPLY_PAULI -- XOR N-bit mask into frame
  • OP_NOISE / NOISE_BLOCK -- stochastic Pauli
  • OP_READOUT_NOISE -- classical bit-flip
  • OP_DETECTOR / POSTSELECT -- parity checks
  • OP_OBSERVABLE -- logical observable
  • OP_EXP_VAL -- expectation value probe

Summary: The HIR at a Glance

Key Design Principles

  • Heisenberg picture -- Clifford gates are absorbed, only non-Clifford + measurements survive
  • Arena-based Pauli masks -- fixed-capacity, zero-copy views, cache-friendly parallel x/z layout
  • 16-byte HeisenbergOp -- 4 per cache line, tagged union payload, type-safe indices
  • Side-tables for variable-length data -- noise channels, detector targets, observable targets
  • Source mapping -- every HIR op traces back to the original Stim source line
