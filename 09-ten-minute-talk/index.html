What Is Clifft?

A simulator that exploits the structure of real QEC circuits

The Problem Clifft Solves

Quantum Error Correction (QEC) circuits are huge

  • Surface codes: thousands of qubits, millions of gates
  • Brute-force statevector: 2n amplitudes — impossible beyond ~40 qubits
  • But >90% of gates are Clifford gates (H, S, CNOT, CZ…)
Clifford gates can be tracked in O(n²) using the stabilizer formalism — no exponential blowup.

Clifft’s Key Insight

Only non-Clifford gates (T gates, rotations) require amplitude vectors. QEC circuits typically have few of these active at any time.

O(2k)
cost per sample, where k = active non-Clifford rank
k ≈ 1–3 for typical QEC circuits (not n = 1000+)

QEC Simulation Essentials

Why QEC circuits have special structure Clifft exploits

What Makes QEC Circuits Special

QEC Circuit Structure

  • Surface codes use thousands of qubits, millions of gates, repeated over many rounds
  • Goal: detect & correct errors by measuring stabilizers (parity checks on qubit groups)
  • To benchmark a decoder, you need millions of Monte Carlo samples (shots) of the noisy circuit
Brute-force statevector simulation costs O(2n) — impossible beyond ~40 qubits. But QEC circuits have exploitable structure.

The 90/10 Split

QEC circuits are almost entirely Clifford gates:

Clifford gates (H, S, CNOT, CZ, SWAP)
~92%
Non-Clifford (T gates, rotations)
~8%

Stabilizer Formalism

Clifford gates map Pauli operators to Pauli operators. Their entire effect can be tracked as an n×2n binary tableau — O(n²) memory, no exponential blowup. Google’s Stim library implements this.

Clifford Absorption — The Key Trick

Traditional Simulator

Every gate updates the full 2n statevector:

q0 —[H]—[CNOT]—[T]—[H]—[S]—M
q1 ————[CNOT]———[CZ]———M
6 operations on 2n statevector

Cost: O(2n) per gate × 6 gates

Clifft’s Approach

Cliffords absorbed into Stim tableau; only T gates touch amplitudes:

[H, CNOT → tableau]
q0 —[T]—M
[H, S, CZ → tableau]
1 operation on v[21] = 2 amplitudes

Cost: O(n²) tableau updates + O(2k) for 1 T gate, k=1

Heisenberg picture: instead of evolving the state forward, Clifft tracks how observables evolve backward through the Clifford tableau. Each non-Clifford op gets a Pauli mask extracted from the inverse tableau — telling the SVM exactly which amplitude indices to touch.

Active Rank k — Why QEC Is Cheap

The dimension that matters

  • Each T gate expands the amplitude vector: k → k+1, doubling its size
  • Each measurement on an active qubit collapses: k → k−1
  • In QEC circuits, T gates are rare and measured quickly → k stays tiny

A d=5 surface code has n > 100 qubits, but peak k ≈ 1–3.
Amplitude vector: 2–8 complex doubles = 16–128 bytes.
Compare brute-force: 2100 ≈ 1030 amplitudes.

k through a QEC round

T gates raise k, measurements lower it — in QEC circuits, they stay close together

CPU Performance Profile

Why It’s Fast on CPU

  • k is tiny: for d=5 surface code, peak k ≈ 1–3
  • v[23] = 8 complex doubles = 128 bytes — fits in L1 cache
  • Frame ops are just XOR on uint64
  • Noise uses exponential skip-ahead — jumps over silent rounds in O(1)
Bottleneck is instruction dispatch, not math. Shots are embarrassingly parallel but each shot is so cheap (~μs) that threading overhead matters.

Typical QEC Cost Breakdown

Frame ops (Clifford) — O(1) per op
~70%
Measurement — collapse + record
~15%
Array ops (T gates) — O(2k)
~10%
Noise — skip-ahead sampling
~5%
For d=5 surface code, ~106 shots/sec on a single CPU core

AMD GPU Execution

HIP backend on MI300X / MI355X

Three-Tier Kernel Strategy

The GPU kernel is selected at launch based on peak_rank — the maximum k in the bytecode:

Mode 1: Thread-Private  (peak_rank ≤ 4)
One thread = one shot. Amplitude array v[≤16] lives entirely in registers.
Grid: ⌈shots/256⌉ blocks × 256 threads. Maximum throughput — no synchronization at all.
Mode 2: Shared-Memory Cooperative  (peak_rank 5–10)
One block = one shot. Amplitude array v[≤1024] in shared memory (LDS on AMD).
Grid: shots blocks × 256 threads. Threads cooperate within a block; __syncthreads() barriers.
Mode 3: Global-Memory Cooperative  (peak_rank 11–19)
Persistent kernel. Amplitude array v[≤524K] in global memory (HBM).
Grid: 2 × CU_count blocks × 256 threads. Work-stealing via atomicAdd on global shot counter.
For QEC workloads, peak_rank ≈ 1–3 → Mode 1 always. This is the fastest path: each thread carries its own tiny amplitude array in registers, processing thousands of bytecode instructions independently.

Parallelization Strategy

Mapping shots to threads, blocks, and wavefronts

The Parallelism Model

Shot-Level Parallelism

Each shot is an independent Monte Carlo sample. No inter-shot communication needed.

Embarrassingly parallel — the ideal GPU workload. Each thread runs the full bytecode program independently with its own RNG seed.

RNG Independence

Each thread seeds its Xoshiro256++ PRNG with:
SplitMix64(base_seed XOR (φ × (shot_id + 1)))
where φ = golden ratio. Guarantees decorrelated streams.

Why This Maps Perfectly to GPUs

✓ What makes it ideal

  • No inter-thread communication in Mode 1
  • Amplitude array fits in registers (k ≤ 4 → 256 bytes)
  • Uniform control flow — all threads execute same bytecode
  • Batching: up to 100M shots per kernel launch
  • Branch divergence only at measurements (random outcomes)

✗ Where it gets harder

  • Higher peak_rank → shared/global memory → sync barriers
  • Mode 3: work-stealing adds atomicAdd contention
  • Large v[] → HBM bandwidth bound, not compute bound
  • Measurement divergence reduces SIMD efficiency

CUDA → HIP: What Changed

Mechanical Translation

The HIP backend is a 1:1 API-name transform of the CUDA backend. All algorithms, data structures, constants, and kernel tiers are identical.

CUDAHIP
cudaMallochipMalloc
cudaMemcpyhipMemcpy
cudaError_thipError_t
__shared____shared__ (same)
__syncthreads()__syncthreads() (same)
atomicAddatomicAdd (same)

Key Differences

Wavefront size: 64 vs 32

AMD uses 64-wide wavefronts vs NVIDIA’s 32-wide warps. But Clifft uses no warp-level primitives — all sync is __syncthreads(), so no algorithmic impact.

FP contraction: off

-ffp-contract=off prevents FMA fusion, ensuring bit-exact reproducibility matching the CPU oracle.

Grid sizing

Mode 3 uses 2 × CU_count blocks.
MI300X: 304 CUs → 608 persistent blocks.
Validation: compare_oracle.py (5σ correctness).
