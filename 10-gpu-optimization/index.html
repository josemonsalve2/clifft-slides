Approaches B + C

Per-Op Kernels and HipGraph

B: Per-Op + C: HipGraph — Both Failed

B+C: Shared Lesson

B: Per-Op Kernels

  • Each opcode is its own GPU kernel
  • State must serialize to global memory between ops
  • 2.4 KB serialized per instruction per shot
  • Bandwidth replaces register pressure as bottleneck
  • Result: +2% thread, −9% coop

C: HipGraph

  • Pre-recorded DAG of kernel launches
  • Single hipGraphLaunch() call
  • Reduces kernel launch overhead to near-zero
  • But execution time dominates — launch overhead was never the bottleneck
  • Result: −1% thread, +1% coop
The bottleneck is sync barriers and register pressure, not launch overhead

GEAK: Warp-Shuffle Reduction

GEAK: coop_reduce2 Before & After

Before: Tree Reduction (8 barriers)

__device__ void coop_reduce2(
    CoopShotState& st,
    double& out0, double& out1) {
  st.red0[threadIdx.x] = out0;
  st.red1[threadIdx.x] = out1;
  __syncthreads();
  for (uint32_t s = blockDim.x >> 1;
       s > 0; s >>= 1) {
    if (threadIdx.x < s) {
      st.red0[threadIdx.x] +=
          st.red0[threadIdx.x + s];
      st.red1[threadIdx.x] +=
          st.red1[threadIdx.x + s];
    }
    __syncthreads(); // 8 barriers
  }
  out0 = st.red0[0];
  out1 = st.red1[0];
}

After: Warp-Shuffle (1 barrier)

__device__ void coop_reduce2(
    CoopShotState& st,
    double local0, double local1,
    double& out0, double& out1) {
  // Phase 1: intra-wavefront (no barriers)
  for (int off = 32; off > 0; off >>= 1) {
    local0 += __shfl_xor(local0, off);
    local1 += __shfl_xor(local1, off);
  }
  // Phase 2: inter-wavefront (1 barrier)
  uint32_t warp_id = threadIdx.x >> 6;
  uint32_t lane    = threadIdx.x & 63;
  if (lane == 0) {
    st.red0[warp_id] = local0;
    st.red1[warp_id] = local1;
  }
  __syncthreads(); // 1 barrier total
  if (threadIdx.x < 4) {
    local0 = st.red0[threadIdx.x];
    local1 = st.red1[threadIdx.x];
    for (int off = 2; off > 0; off >>= 1) {
      local0 += __shfl_xor(local0, off);
      local1 += __shfl_xor(local1, off);
    }
  }
}

Pitfalls & Dead Ends

When optimizations make things worse

Pitfall: Scatter-Bits Lookup Table

Expected

  • Replace ~15 SALU instructions with 1 LDS load
  • Precomputed 16KB table in shared memory
  • Target: reduce SALU/VALU ratio at rank=10
  • Predicted: -20% to -30% SALU stalls

Actual

  • VGPRs: 108 → 128 (table index variables)
  • Throughput: -3.8%
  • Extra VGPR pressure caused scheduling stalls
  • LDS bank conflicts on random access pattern
Root cause: register pressure increase from index tracking outweighed any SALU savings. The SALU instructions were cheap; the new VGPR spills were not.

Also tried: 16KB precomputed table + 8KB table — both showed the same pattern. The fundamental issue is that scatter-bit indices themselves consume registers.

Pitfall: MFMA Matrix Acceleration — 45–70x Slower

Why MFMA Fails for Clifft

  • Gate matrices are 2×2 or 4×4 complex
  • MFMA minimum K dimension = 4 (real)
  • Complex gates require 4 real MFMAs each
  • K=2 for 2×2 complex → below MFMA minimum
  • Gather/scatter overhead: ~80 cycles per gate
MFMA is designed for large matrix multiplications (128×128+). Gate matrices in QEC circuits are tiny — the overhead dwarfs any computation benefit.

(not applied)

MFMA Applicability Threshold

MFMA pays off when: matrix K ≥ 16 (real), batch size ≥ 16, no gather/scatter needed. Clifft's 2×2 and 4×4 gates meet none of these criteria.

Pitfall: LDS Tiling — Extra Barriers Destroyed Gains

Expected

  • Replace random HBM accesses with coalesced tile loads
  • Stage amplitude data in LDS before sweep
  • Reduce HBM bandwidth pressure at rank=10
  • Expected: -20% to -40% memory latency

Actual Results

Root cause: The coop kernel already stores amplitudes in LDS. Adding tiling introduced 3 new __syncthreads() barriers per sweep where zero existed before. Net effect: catastrophic slowdown.

__noinline__: The One Micro-Optimization That Worked

What Worked

  • Extract coop_u2_sweep and coop_u4_sweep with __noinline__
  • VGPRs: 128 → 108
  • Prevents inlining of large sweep bodies
  • Compiler can now schedule outer loop registers independently

Critical Constraint

Must use raw GpuComplex* pointers, NOT CoopShotState& references.

HIP rejects __noinline__ on functions containing __syncthreads() when passed struct references.

5 Key Findings

1. Compiled megakernel is a REGRESSION, not an improvement

Approach A shows -50% to -69% throughput loss after rigorous benchmarking (20 runs, GPU warmup, fixed pipeline). The original +5-18% claim was measurement noise from a non-functional --hybrid flag. The --offload-device-only path produces inferior code.

2. peak_rank determines throughput by 330x

Tier 1 (rank 0–4): 48M shots/s. Tier 2 (rank 5–10): 5.6M. Tier 3 (rank 11–19): 145K. Circuit selection matters more than kernel optimization.

3. The HIP compiler is excellent at micro-optimizations

All 7 manual micro-optimizations in Approach F achieved 0% gain. ROCm clang++ -O3 already does them. Don't fight the compiler.

4. HIPRTC vs AOT matters: -14%

Runtime JIT compilation loses 14% throughput vs offline clang++. Always use AOT with disk-cached .hsaco binaries.

5. Synchronization is the #1 coop bottleneck

Warp-shuffle reduction eliminates 7 of 8 barriers per measurement. +40% throughput. Synchronization cost, not compute, dominates cooperative kernels.

Priority Stack: What to Do (and Not Do)

High Priority (apply now)

1. GEAK warp-shuffle — already applied
+40% coop tier. Zero downsides. SVM+GEAK is the baseline going forward.
2. Compiled kernel — DO NOT USE (regression)
-50 to -69% throughput regression. The --offload-device-only compilation path produces inferior code. Needs fundamental rethinking before revisiting.
3. __noinline__ coop sweeps
VGPRs 128→108 via targeted extraction of coop_u2_sweep and coop_u4_sweep.

Medium Priority (circuit-dependent)

4. Heuristic split — sawtooth circuits only
3–5x potential when circuits have genuine low-k regions. Verify active_k profile first.
5. NUMA per-XCD — rank≥15 only
Reduces cross-XCD latency for global-coop tier. Cannot break HBM bandwidth ceiling.

Do Not Pursue

  • Compiled megakernel (-50 to -69% regression)
  • MFMA acceleration (45–70x slower)
  • LDS tiling (-66% on coop tier)
  • Scatter-bits LUT (-3.8%, VGPR pressure)
  • Any HIPRTC JIT path (-14%)

What's Next

Near-Term (actionable)

  • Heuristic split on sawtooth circuits — identify QEC circuits with genuine EXPAND→low-k→EXPAND patterns, apply split, verify 3–5x gain
  • NUMA per-XCD for rank≥15 — implement XCD-affinity workgroup routing for global-coop tier
  • Upstream GEAK warp-shuffle — contribute to clifft upstream as a device-agnostic improvement

Hardware Roadmap

  • MI350X (CDNA4) — same ISA for __shfl_xor, same optimizations apply
  • Higher HBM bandwidth — rank=19 ceiling will rise proportionally
  • Larger LDS — may enable more aggressive coop strategies

Open Questions

  • Can warp-shuffle extend to inter-block reductions?
  • Is there a circuit class where MFMA becomes viable (k≥16)?
  • Can a non---offload-device-only compilation path recover the compiled kernel approach?
The architecture is stable. GEAK is the breakthrough. Next step is finding sawtooth circuits to unlock the 3–5x split gain.
